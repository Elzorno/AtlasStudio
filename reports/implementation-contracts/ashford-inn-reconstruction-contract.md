## 9. Implementation Guidance

**Authoritative requirements** (from `IMP-HOM-020` / `SCR-HOM-ASH-004` - unchanged, non-negotiable, reproduced from the original contract's Section 9 since they still govern this build):

- Map size ~16x12 tiles (already correct; no change needed).
- Reception counter near the entrance, on the Innkeeper's interaction axis; blocks on every side except that side.
- Two to three bed alcoves along a back or side wall, implying private rooms without building them.
- A warm-stone vent or small hearth, per Ashford's established village motif.
- `EVT-HOM-032`, `TRN-HOM-031`/`TRN-HOM-032`, and the three story states - all already correctly built per `OBS-ASHFORDINN-001`'s event inventory; **preserve as-is, do not regenerate**.
- Tileset family `Inside` (already correct; confirmed by `OBS-ASHFORDINN-001` against `Tilesets.json` - no change needed).
- Explicit non-goals: no shop, no inventory, no gold cost on Rest; no automatic final map construction (`BUILD-0043`'s NO GO applies to this reconstruction pass exactly as it applied to the original build).

**Pattern-derived guidance** (from `Interior` only - unchanged from the original contract's Section 9):

- Compact architectural shell inside a void/darkness border (already correct - `OBS-ASHFORDINN-001` confirms the border/void tiling matches this convention).
- Visible threshold tile at the exit, player spawn one tile inside the doorway on entry (already correct; not affected by the three defects below).

**Reconstruction-specific requirements** (NEW - from `academy/observations/OBS-ASHFORDINN-001.json` and `academy/reports/ashford-inn-revisit-001.md`; authoritative for this pass because each traces directly to `IMP-HOM-020` Acceptance Criterion 5, "landmarks... visually distinct," and to `bridges/rpg-maker-mz/map-quality-standard.md`'s "believable as a polished RPG Maker MZ sample map" bar):

1. **Fireplace, full assembly required.** The fireplace at map tiles (8,2)-(9,3) currently uses only sheet columns 14-15 of `Inside_B.png`'s three-column fireplace object (rows 7-8, columns 13-15), omitting column 13 (the object's left pillar). Rebuild this landmark using all three columns (13-15) so both pillars are present - this will require widening the placed footprint by one column (to roughly (7,2)-(9,3), adjusted to fit the room's existing layout) rather than compressing a 3-wide object into a 2-wide space.
2. **Bed alcoves, complete tile sets required.** The three bed alcoves at (2,4)-(2,5), (5,4)-(6,5), and (11,4)-(12,5) each use only their bed(row5)/curtain(row6) tiles, omitting the matching shelf(row4)/table(row7) tiles that complete each style elsewhere on `Inside_B.png`. Every alcove placed must include its full four-row set (shelf, bed, curtain, table) from the same sheet column group it draws from. **Implementation recommendation, not authoritative:** consider using one consistent bed style (a single column group) across all alcoves rather than three different styles side by side, per `GRD-ASHFORDINN-002`'s composition finding - `IMP-HOM-020` does not mandate a specific style or require uniformity, so this is a judgment call for the implementer, not a rule.
3. **No unsupported wall-mounted tiles.** The picture-frame tiles (world map, painting, portrait) currently placed at floor level (y=7, x=6-10) have no cabinet/base tile beneath them (row 2 of the same sheet columns). `IMP-HOM-020` does not require wall art at all, so the simplest compliant fix is to remove these tiles entirely; if the implementer chooses to keep them for atmosphere, they must be paired with their matching base tile, per the sheet's intended pairing. Either resolution is acceptable; leaving them floor-placed without a base is not.
4. **"Rocks" - open item, watch for it during the session.** `OBS-ASHFORDINN-001` found no rock, stone-debris, or rubble tile or event graphic anywhere on this map. This is not resolved by this contract. During the reconstruction session, the implementer should specifically look for anything in the live RPG Maker editor view that could read as "rocks" beyond what static tile-ID inspection surfaced (see Section 12) and note it if found, rather than assuming Findings 1-3 above fully account for the original rejection reason.

**Implementation recommendations** (non-binding, unchanged from the original contract): the optional decorative flame and optional passing-traveler NPC remain available per `ashford-inn-pattern-aware-contract.md` Section 9, not required by this reconstruction.

## 14. Recommended Implementation Agent

**Human (Production Director or delegated map author), in the RPG Maker MZ editor directly** - not an automated or agent-driven edit.

Rationale: `IMP-HOM-020` and `BUILD-0043` (`../TheLastSwordProtocol-Atlas`'s formal NO GO for automatic final map construction) already required manual/guided build discipline for the *first* attempt at this exact map, and that attempt still produced the three defects this contract exists to fix. A second automated or semi-automated attempt at precision tile placement carries the same risk that produced the rejection the first time, with less excuse the second time. Per `studio/agent-assignment-model.md`'s task classification (this work is `implementation`/`repetitive_edit`, asset-placement craft judgment, not architecture or creative design), no AI provider scores above a `limited`/reviewed assignment for this category of hand tile-placement work without a human driving the actual editor session.

**Codex** (`implementation-engineer`) may assist with the Section 12 validation script (a fresh Academy observation pass is exactly the kind of read-only, deterministic JSON-inspection task `studio/agent-roles.md` recommends Codex for) but should not perform the tile edits themselves.
