# Atlas Import Architecture

## Purpose

`work-orders/WO-0018-repository-authority.md` established that `TheLastSwordProtocol-Atlas` is the authoritative repository for The Last Sword Protocol's story, canon, work orders, design, and production intent, and that AtlasStudio must not act as a competing source of truth. This document designs how AtlasStudio imports and understands that information so it can plan, validate, orchestrate, review, schedule, and bridge to implementation - without ever becoming a second canon.

This is a design document. It describes an architecture; it does not implement one. No tooling described here exists yet (see "Future Tooling" for what should be built later, by a separate work order).

## The Governing Constraint

Every design choice below serves one rule, restated from `studio/governance/repository-authority.md`:

```text
Atlas
  ↓
AtlasStudio
  ↓
Implementation
  ↓
Validation
```

Information flows downhill. AtlasStudio may read from Atlas. AtlasStudio may never write to Atlas, except through an explicit, human-approved proposal (per `repository-authority.md`'s change-flow model) - never automatically, and never as a side effect of import.

## Discovery

### Primary Discovery Source: The Atlas Export Contract

`TheLastSwordProtocol-Atlas` already produces a structured export designed for exactly this purpose: `atlas-exports/home-island.json`, generated by `atlas-tools/cli/atlas.py export home-island`. Direct inspection of this file shows it already declares its own import contract:

```json
{
  "schema_version": "1.0.0",
  "generated_at": "2026-07-04T14:20:41+00:00",
  "source": { "git_branch": "main", "git_commit": "bc3598b", "repository": "TheLastSwordProtocol-Atlas", "paths": { "...": "..." } },
  "contract": {
    "atlas_owns": ["Atlas IDs", "screen, transfer, and event registries", "implementation packet requirements", "database-ready rows and placeholder policies", "production readiness decisions"],
    "game_owns": ["RPG Maker map IDs and event numbers", "runtime JSON files", "final map coordinates", "asset files and project settings", "implementation reports"],
    "read_only_import": true
  },
  "home_island": { "screens": [...], "transfers": [...], "events": [...], "tilesets": [...], "animations": [...], "combat_database": {...}, "trial_mechanics": {...}, "common_event_candidates": [...], "production_readiness": {...} }
}
```

This is a gift AtlasStudio should not ignore: Atlas has already designed and labeled its own export as `read_only_import: true`, with an explicit `atlas_owns`/`game_owns` split. **AtlasStudio's import pipeline should treat this export as its primary discovery source**, not re-derive the same information by parsing hundreds of individual Markdown files directly. Discovery means: locate the newest `atlas-exports/*.json` file, read its `schema_version` and `source.git_commit`, and treat its contents as the current snapshot of importable state.

### Secondary Discovery Sources (Not Yet Export-Covered)

The Home Island export covers screens, transfers, events, tilesets, animations, combat database, and trial mechanics - it does not currently cover work orders, characters as standalone entities, quests, or dialogue references. Until Atlas extends its export to include these (a proposal AtlasStudio may offer, per `repository-authority.md`, but not implement unilaterally), discovery must also read:

- `atlas/workorders/*.md` and `atlas/planning/workorder_queue.json` for work orders and production state.
- `atlas/docs/05_Characters/*.md` for character objects (frontmatter: `object_id`, `atlas_id`, `object_type: NPC`, `relationships`).
- `atlas/docs/03_Story/Quests/` for quest objects.
- `../TheLastSwordProtocol-Game/map_ownership.json` for map ownership state.
- `../TheLastSwordProtocol-Game/data/System.json` and `Tilesets.json` for engine-level ground truth (start position, tileset roster) that no Atlas export currently carries.

Every secondary source read this way should still be normalized into the same entity shape as export-derived data (see `imported-entity-model.md`), so downstream tooling never needs to know which discovery path produced a given entity.

### Discovery Is Read-Only, Always

No discovery step writes to any file outside AtlasStudio. Discovery over Markdown frontmatter uses the same read-only pattern AtlasStudio's own tools already use elsewhere (`tools/atlas_graph/validate_graph.py` reads JSON without writing it; the same discipline applies here to reading Markdown and JSON in sibling repositories).

## Parsing

Two distinct parsers are needed, because Atlas represents information in two different shapes:

1. **Export JSON parser** - straightforward JSON parsing of `atlas-exports/*.json`, keyed by the `home_island` (or future per-region) payload structure. This is the preferred path whenever a field is export-covered.
2. **Frontmatter object parser** - Atlas's Markdown documents carry a consistent YAML frontmatter block (`object_id` or `atlas_id`, `title`, `object_type`, `status`, `version`, `canonical`, `owner`, `implementation_status`, `relationships` or `dependencies`/`related`), confirmed by direct inspection of character, screen, and implementation packet documents. A single frontmatter parser can extract this consistently across every object type, since the shape does not vary by `object_type`.
3. **Work order queue parser** - `atlas/planning/workorder_queue.json` is a structured array of candidate/active/completed work order records with their own fields (`title`, `slug`, `active`, `completion_note`, `milestone`, `priority`, `rationale`, `goal`, `files_to_inspect`), distinct from both the export and the frontmatter shape. Individual `atlas/workorders/WO-NNNN-*.md` files, notably, do **not** carry frontmatter themselves (confirmed by direct inspection of `WO-0036`) - their structured metadata lives in the queue JSON, not the file. Parsing must read both and join them by work order ID.

Parsing never mutates source files. Parsing failures (malformed frontmatter, missing required fields, an export whose `schema_version` the importer does not recognize) should be reported, not silently skipped or guessed at - see Conflict Detection.

## Normalization

Every parsed record - whether from the export JSON, frontmatter, or the work order queue - is normalized into one common internal shape before AtlasStudio touches it: the Imported Entity Model (`imported-entity-model.md`). Normalization means:

- Mapping Atlas's `object_type` values (`NPC`, `Screen`, `ImplementationPacket`, and so on) to AtlasStudio's imported entity `kind` field.
- Preserving every source-identifying field untouched (see Traceability below) rather than discarding them once normalized.
- Never inferring a fact that Atlas did not state. If a field is absent in the source, it is absent in the imported entity - normalization does not fill gaps with assumptions.

## Traceability

Every imported entity must carry its provenance forward, permanently. At minimum:

- `source_repository` - always `TheLastSwordProtocol-Atlas` today, but stated explicitly rather than assumed, since a future import source is possible.
- `source_path` - the exact file path the record came from (a Markdown file path, or the export JSON path plus a pointer into it, e.g. `atlas-exports/home-island.json#/home_island/screens/0`).
- `source_identifier` - the Atlas object ID or work order ID (`SCR-HOM-ASH-001`, `WO-0036`, `NPC-ELA-001`).
- `source_commit` - the `source.git_commit` value from the export, or a direct `git rev-parse HEAD` read at discovery time when reading frontmatter/queue files directly, so every import is pinned to an exact point in Atlas's history.
- `imported_at` - the timestamp AtlasStudio performed the import, distinct from `generated_at` (when Atlas produced the export).
- `imported_version` - a simple monotonic counter or the AtlasStudio import tool's own version, so repeated imports of the same source state are distinguishable in history.

No imported entity is ever presented, queried, or reported without these fields visibly attached. This is the mechanism that makes "traceable back to Atlas" concrete rather than aspirational.

## Versioning

Two independent version concepts must not be conflated:

- **Source version** - the export's `schema_version` and `source.git_commit`, or a direct commit hash read from Atlas's own git history for non-export sources. This tells AtlasStudio *which state of Atlas* a given imported entity reflects.
- **Import version** - AtlasStudio's own `imported_version` counter, incrementing each time the same source entity is re-imported, even if the source itself has not changed. This tells AtlasStudio *how many times* it has looked at that entity, useful for debugging the import pipeline itself, separate from tracking Atlas's own history.

A schema version mismatch (the export declares a `schema_version` AtlasStudio's importer does not recognize) is a hard stop, not a best-effort parse - see Conflict Detection.

## Conflict Detection

"Conflict" here means a mismatch AtlasStudio's import pipeline discovers, not a creative disagreement to resolve (that is the conflict-analysis process WO-0018 already performed once, and any future recurrence of that kind of conflict should be escalated the same way - to a human, via a governance work order, not auto-resolved). Import-time conflict detection specifically means:

- **Renamed or re-identified entities:** an Atlas ID AtlasStudio has previously imported no longer resolves, or a `title` changed while the ID stayed the same (expected and fine, per the Canonical ID Registry's "IDs are stable, names can change" rule) versus an ID that disappeared entirely (needs reporting - see `synchronization-strategy.md`).
- **Schema drift:** the export's `schema_version` changed in a way the importer does not have a mapping for.
- **Cross-source disagreement:** the export says one thing about a screen (e.g., its `implementation_status`) while a directly-read frontmatter file says another, implying the export is stale relative to the live repository state.
- **Ownership contradiction:** an entity the export's `contract.atlas_owns` lists is found to have a competing definition inside `TheLastSwordProtocol-Game` (for example, a map name mismatch between `rpg_maker_map_name` in the screen frontmatter and the actual `MapInfos.json` entry).

Every detected conflict is reported to a human-readable log, never silently resolved by the importer picking a side. This mirrors the exact principle `studio/governance/atlas-principles.md` states for "deterministic systems before opaque AI" - conflict detection is a deterministic check, and its output is a report, not a decision.

## Where Imported Data Lives

Imported entities are stored under `bridges/atlas/imported/` (a future location, not created by this design-only work order), clearly separated from `projects/the-last-sword-protocol/graph/`, which `WO-0018` already marked historical and non-authoritative for this project. This separation is deliberate: the imported cache must never be mistaken for a new AtlasStudio-authored canon. It is a mirror, timestamped and sourced, always regenerable from Atlas, never hand-edited. Storing it in the same JSON node/edge shape AtlasStudio's existing graph tooling already understands (`tools/atlas_graph/validate_graph.py`, `query_graph.py`, `diff_graph.py`) is a deliberate reuse choice - see `synchronization-strategy.md` for how the existing Graph Diff Engine applies directly to imported-state comparison.

## Future Tooling (Recommended, Not Implemented)

This work order is documentation only - nothing below is built here. It is named specifically so a future implementation work order has concrete targets rather than a vague mandate.

- **`atlas_import.py`** - performs first import and re-import: discovery (export JSON preferred, frontmatter/queue fallback), parsing, normalization into the `imported-entity-model.md` shapes, and writing the `bridges/atlas/imported/` cache. Should be read-only against both sibling repositories and write-only against AtlasStudio's own cache directory.
- **`atlas_sync.py`** - runs the incremental-update comparison in `synchronization-strategy.md`: diffs newly-discovered state against the existing cache, stamps `imported_version`, and produces the synchronization report (added/changed/`not_found_at_last_sync`/conflicts). Likely shares comparison logic with `atlas_diff.py` rather than duplicating it.
- **`atlas_diff.py`** - given two points in the imported cache's own history (or a cache state versus a fresh discovery pass), reports what changed, reusing the comparison and reporting patterns already proven in `tools/atlas_graph/diff_graph.py`. This is the tool a human runs to answer "what changed in Atlas since I last looked," independent of whether a full sync has been triggered.
- **`atlas_status.py`** - a read-only reporting tool (in the spirit of `tools/atlas_doctor/doctor.py`) summarizing the imported cache's health: how many entities of each kind, how stale the last import is relative to Atlas's current commit, how many `not_found_at_last_sync` entries are outstanding, and how many implementation-handoff records are in each status.

None of these tools exist yet. Building them is future, separately-scoped implementation work, and should itself go through AtlasStudio's normal work-order process rather than being built as a side effect of this design.

## What This Architecture Does Not Do

It does not import automatically on a schedule - cadence is deferred to whoever builds `atlas_sync.py`. It does not resolve any conflict it detects. It does not modify `TheLastSwordProtocol-Atlas` or `TheLastSwordProtocol-Game`. It does not implement any parser, normalizer, or storage format described above - see `bridges/atlas/imported-entity-model.md`, `synchronization-strategy.md`, and `implementation-handoff.md` for the remaining design.
